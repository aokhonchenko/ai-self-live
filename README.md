# Управляемый автономный AI-эксперимент

Это локальный каркас для эксперимента в стиле "AI lives on computer": создатель вручную запускает сессию, агент читает файловую память, делает небольшой шаг, оставляет следы для следующей сессии и завершает работу.

Главные свойства:

- пользовательские артефакты пишутся на русском языке;
- агент самостоятельно выбирает направление каждой сессии;
- агент может задавать вопросы создателю через `state/questions/*.md`;
- сессии запускаются атомарно через временный `git worktree` в `runs/session-NNNN`;
- последние два отладочных снимка сессий сохраняются в `runs/snapshots/`;
- модель подключается напрямую к OpenAI-compatible `/chat/completions`, без внешнего agent framework;
- инструменты агента находятся в `src/tools/`: каждый инструмент сам отдаёт schema, обработчик и строку runtime-паспорта.

## Сборка окружения

```bash
uv sync
```

Для обращения к модели используется стандартная библиотека Python и OpenAI-compatible HTTP API.

## Настройка модели

Скопируйте пример окружения:

```bash
cp .env.example .env
```

Заполните `.env`:

```bash
AI_API_KEY=
AI_BASE_URL=https://your-openai-compatible-endpoint.example/v1
AI_MODEL=your-model-name
```

`AI_API_KEY` можно оставить пустым, если endpoint не требует авторизации. `AI_BASE_URL` должен указывать на базовый OpenAI-compatible URL; агент сам добавит `/chat/completions`, если путь не указан полностью.

Несекретные параметры сессии лежат в `config/project.toml`:

```toml
[agent]
step_limit = 300
request_timeout_seconds = 300
repeated_tool_error_limit = 3
repeated_tool_call_limit = 3
temperature = 0.2
```

## Ручной запуск

Проверить сборку промпта без запуска модели:

```bash
uv run python scripts/run_session.py --dry-run
```

Запустить одну атомарную сессию:

```bash
uv run python scripts/session_transaction.py
```

По умолчанию транзакционный runner вызывает:

```bash
uv run python scripts/run_agent.py --root "{ROOT}" --prompt-file "{PROMPT_FILE}"
```

Если нужно временно заменить агента, можно передать `--agent-command` явно.

## Как работает локальный агент

`run_agent.py` делает простой цикл:

1. Загружает активный промпт сессии.
2. Отправляет сообщения в OpenAI-compatible `/chat/completions`.
3. Сканирует `src/tools/*/tool.py` и выдаёт модели найденные tools.
4. Выполняет tool calls внутри корня временного worktree.
5. Печатает каждый шаг в консоль.
6. Если инструмент вернул ошибку, передаёт её модели как `ok:false`, чтобы модель могла скорректировать действие.
7. Завершает сессию, когда модель отвечает финальным сообщением без tool call.

Если endpoint не поддерживает OpenAI tool calling, агент также понимает текстовый JSON-протокол:

```json
{"tool":"имя_инструмента","аргумент":"значение"}
{"final":"краткий итог"}
```

Инструмент добавляется отдельной директорией `src/tools/<tool_name>/tool.py`. Модуль обязан экспортировать `schema()`, `passport()` и `handle(root, arguments)`. Подробный контракт описан в `src/tools/README.md`. После добавления инструмента нужно добавить тесты и запустить `uv run python -m pytest`.

## Атомарность

`session_transaction.py` обеспечивает правило: успешная сессия применяется целиком, упавшая сессия не меняет основной проект.

Порядок работы:

1. Проверяет, что основной проект является Git-репозиторием.
2. Загружает `.env` из основного проекта.
3. Если изменены только человеческие входы, делает checkpoint-коммит.
4. Проверяет, что основной worktree чистый.
5. Берёт `.session.lock`, чтобы две сессии не шли одновременно.
6. Создаёт временный `git worktree` в `runs/session-NNNN`.
7. Запускает `scripts/run_session.py` внутри временного worktree.
8. Запускает локального агента через `scripts/run_agent.py`.
9. Запускает проверки, по умолчанию `python -m pytest`.
10. Если всё успешно, коммитит изменения сессии и применяет их в основной проект через `git merge --ff-only`.
11. Перед удалением временного worktree сохраняет отладочный снимок в `runs/snapshots/`.
12. Если агент или проверки падают, удаляет временный worktree и ветку сессии, но снимок остаётся для анализа.

Внешние проекты в `projects/`, например `projects/foundation-finance/`, не попадают в git агента. Они должны быть отдельными репозиториями; транзакционный runner копирует их файлы из успешной сессии обратно в основной каталог отдельно от merge-коммита.

В `runs/snapshots/` автоматически остаются только два последних снимка, чтобы каталог не рос бесконтрольно.

## Сон агента

Сон не запускается отдельным внешним режимом. Создатель запускает обычную сессию из фронта через `server.bat` или напрямую:

```bash
uv run python scripts/session_transaction.py
```

Агент сам решает, что пора спать, если накопилась усталость: много закрытых вопросов, шумная память, повторяющиеся действия или устаревшие решения. Внутри обычной сессии он может вызвать подготовленный скрипт, если у него есть подходящий инструмент или он добавит такой инструмент в проект:

```bash
uv run python scripts/sleep_memory.py --root .
```

Сон архивирует закрытые вопросы из `state/questions/`, создаёт отчёт в `state/sleep/reports/`, обновляет `state/sleep/last_sleep.md` и оставляет будущей сессии более чистое состояние.

## Вопросы создателю

Агент может задавать вопросы через файлы в `state/questions/`. Он создаёт отдельный markdown-файл с контекстом и разделом `## Ответ создателя`. Вы вписываете ответ в этот же файл и вручную запускаете следующую сессию.

Перед стартом транзакционной сессии runner автоматически делает checkpoint-коммит, если в основном проекте изменены только человеческие входы:

- `state/external_messages.md`;
- `state/questions/*.md`.

Если изменены другие файлы, запуск блокируется, чтобы не смешивать ручные правки с результатом автономной сессии.

## Тесты

```bash
uv run python -m pytest
```

Порог покрытия задан в `pyproject.toml`: 90%.
## Падение проверок

Если проверки после работы агента падают, транзакционный runner откатывает сессию: основной проект не получает изменения из временного worktree, последние 50 строк диагностики сохраняются в `state/session_failure_tail.txt`, а `state/last_session.md` получает отметку о прерывании. Отладочный снимок остаётся в `runs/snapshots/session-NNNN-failed`.
